# LeStudio — Wireframe Design Brief

**목적**: 이 문서는 와이어프레임 디자이너에게 "앱이 어떤 기능을 제공하고, 각 기능에서 무엇을 보여주며, 사용자가 무엇을 할 수 있는지"를 전달합니다.
**원칙**: 현재 시각 디자인에 대한 정보를 의도적으로 배제합니다. 디자이너의 자유도를 최대화하기 위함입니다.

> **⚠️ 중요: 화면 구성(Information Architecture)은 디자이너의 영역입니다.**
> 이 문서는 "기능 도메인"을 나열합니다. 이 기능들을 몇 개의 화면/탭으로 나눌지,
> 어떤 기능을 합치거나 분리할지는 전적으로 디자이너가 결정합니다.
> 현재 구현의 탭 구조에 구애받지 마세요.

최종 갱신: 2026-03-01

---

## 1. 제품 한 줄 정의

> 로봇 팔을 설정하고, 원격 조작하고, 데이터를 모아 AI를 학습시키는 **데스크톱 웹 워크벤치**.

### 타겟 사용자
- 로봇 연구자 (대학원생, 연구소)
- 취미 로봇 빌더 (메이커)
- LeRobot 프레임워크 사용자

### 기술 환경
- **플랫폼**: 데스크톱 웹 (Chrome/Firefox, Linux)
- **보조 지원**: 태블릿 (가로 모드), 모바일 (최소 지원)
- **테마**: 다크 모드 + 라이트 모드 전환 필수

---

## 2. 기능 흐름 (Functional Flow)

사용자는 아래 순서로 작업을 진행합니다. 단, 각 단계를 별도의 화면으로 만들지,
여러 기능을 하나의 화면에 통합할지는 디자이너가 결정합니다.

```
하드웨어 준비          운영              데이터           ML
──────────────      ──────           ──────         ──────
① 시스템 상태    →   ⑤ 원격 조작  →   ⑦ 데이터셋  →  ⑧ AI 학습
② 디바이스 매핑  →   ⑥ 에피소드 녹화    관리         ⑨ 정책 평가
③ 모터 설정
④ 캘리브레이션
```

- **왼→오 순차 진행**이 기본이지만, 각 기능은 독립적으로도 접근 가능해야 합니다.
- ⑤~⑥은 동일한 하드웨어(카메라 + 로봇 팔)를 사용하므로 설정이 공유됩니다.
- ②~④는 모두 "하드웨어 준비" 단계로, 한 번 완료하면 반복할 필요가 적습니다.

## 3. 글로벌 UI (모든 화면에 항상 존재)

### 3.1 앱 쉘 구조

```
┌─────────────────────────────────────────────────┐
│  Header: [☰] [로고/홈] ···· [프로필] [테마] [HF] [링크] [WS상태] │
├──────┬──────────────────────────────────────────┤
│      │                                          │
│ Side │          Main Content Area               │
│ bar  │          (기능별 콘텐츠)                      │
│      │                                          │
│      │                                          │
├──────┴──────────────────────────────────────────┤
│  Console Drawer (접기/펼치기 가능, 드래그 리사이즈)       │
│  [프로세스 선택] [로그 영역] [stdin 입력] [복사/클리어]    │
└─────────────────────────────────────────────────┘
```

### 3.2 헤더 (상단 바)
| 요소 | 설명 |
|---|---|
| 햄버거 메뉴 | 모바일에서 사이드바 토글 |
| 로고/홈 버튼 | 클릭 시 홈(시스템 상태)으로 이동 |
| 프로필 셀렉터 | 설정 프로필 저장/불러오기/가져오기/내보내기/삭제 |
| 테마 토글 | 다크 ↔ 라이트 전환 |
| HF 인증 상태 | Hugging Face 로그인 여부 표시 |
| WS 연결 상태 | 서버 연결 상태 (연결됨/불안정/끊김) — 색상 점 |

### 3.3 네비게이션 (좌측)
- **기능 그룹별 메뉴** — 아래는 기능의 논리적 분류이며, 실제 메뉴 구조와 라벨은 디자이너가 결정:
  - 하드웨어 준비: 시스템 상태, 디바이스 매핑, 모터 설정, 캘리브레이션
  - 운영: 원격 조작, 에피소드 녹화
  - 데이터: 데이터셋 관리
  - ML: AI 학습, 정책 평가
- **각 메뉴 항목 옆에 상태 뱃지** 표시 가능:
  - 프로세스 실행 중 (Running)
  - 주의 필요 (Warning)
  - 의존성 누락 (Missing Dep)
  - 디바이스 필요 (Needs Device)
  - Root 권한 필요 (Needs Root)
- **반응형 동작**:
  - 데스크톱: 항상 펼쳐진 사이드바
  - 태블릿: 아이콘만 표시하는 레일
  - 모바일: 햄버거 메뉴로 열리는 서랍

### 3.4 콘솔 드로어 (하단 고정)
| 요소 | 설명 |
|---|---|
| 프로세스 셀렉터 | 현재 실행 중인 프로세스 전환 (teleop, record, train 등) |
| 실행 상태 바 | 활성 프로세스별 진행 표시 + 해당 기능 이동 버튼 + Stop 버튼 |
| 로그 영역 | stdout/stderr 실시간 스트림, 종류별 색상 구분 (일반/경고/에러/번역) |
| stdin 입력 | 실행 중인 프로세스에 텍스트 명령 전송 |
| 복사/클리어 | 로그 전체 또는 최근 N줄 복사, 로그 클리어 |
| 리사이즈 | 드래그로 콘솔 높이 조절 |

### 3.5 토스트 알림
- 화면 모서리에 표시되는 일시적 메시지
- 종류: 성공 / 에러 / 정보
- 자동 사라짐, 호버 시 일시정지, 수동 닫기

### 3.6 글로벌 키보드 단축키
| 키 | 동작 |
|---|---|
| → (Right Arrow) | 녹화: 다음 에피소드 (Save) |
| ← (Left Arrow) | 녹화: 에피소드 폐기 (Discard) |
| Esc | 녹화: 녹화 종료 (End) |

- Space 단축키는 LeRobot 원본 키 처리 정합성을 위해 현재 와이어프레임 범위에서 의도적으로 제외한다.

### 3.7 데스크톱 알림
- 프로세스 완료/에러 시 브라우저 Notification API로 알림
- 화면이 비활성 상태일 때 특히 유용
- frontend 덮어쓰기 준비를 위해 wireframe_test에서도 동일한 알림 흐름(Record 종료, Train 완료, 프로세스 에러)을 선반영한다.

## 4. 기능 도메인 명세

> 아래 9개 도메인은 **기능의 논리적 분류**입니다.
> 각 도메인을 별도 화면으로 만들지, 합치지, 나눌지는 전적으로 디자이너가 결정합니다.
> 예: "시스템 상태"와 "디바이스 매핑"을 한 화면에 통합해도 되고,
> "데이터셋 관리"를 여러 화면으로 분리해도 됩니다.

### 4.1 시스템 상태 모니터링

**목적**: 연결된 하드웨어와 시스템 상태를 한눈에 확인하는 대시보드 기능.

#### 보여줄 정보

**A. 전체 준비 상태 판정**
- 상태 뱃지: "Ready" 또는 "Action Needed"
- 만약 준비 안 됨: 구체적 이유 목록 (예: "카메라 미연결", "매핑 미완료")
- 각 이유별 빠른 이동 링크 (해당 기능으로) — 칩(chip) 형태의 뱃지로 표시

**B. Hugging Face 인증**
- Hub 로그인 상태: 사용자명 또는 "Not Logged In"
- Hub 인증 관리 기능으로 이동하는 액션

**C. 카메라 목록**
- 감지된 카메라별:
  - 이름 (심볼릭 링크 또는 디바이스명)
  - 경로, 포트 번호, 모델명
  - 매핑 상태 뱃지: "linked" 또는 "no link"
- 빈 상태: "카메라가 감지되지 않았습니다. USB 카메라를 연결하고 새로고침하세요."

**D. 팔 포트 목록**
- 감지된 로봇 팔별:
  - 이름 (심볼릭 링크 또는 디바이스명)
  - 경로, 시리얼 번호
  - 매핑 상태 뱃지
- 빈 상태: 카메라와 동일한 구조의 안내 메시지

**E. 프로세스 상태**
- 6개 프로세스 각각의 실행 상태:
  - Teleop / Record / Calibrate / Motor Setup / Train / Eval
  - 상태: "running" (초록 점 + 펄스 애니메이션) 또는 "idle" (회색 점)

**F. 시스템 리소스**
- CPU 사용률: 퍼센트 + 바
- RAM: 사용량 / 전체 + 바
- 디스크 (home): 사용량 / 전체 + 바
- LeRobot 캐시 크기: 용량 표시
- 각 바에 심각도별 색상 (정상/주의/위험)
- 로드 실패 시: 에러 메시지 + 재시도 버튼

**G. 세션 히스토리**
- 시간순 이벤트 목록 (캘리브레이션, 녹화, 학습, 평가)
- 각 항목: 이벤트 타입 + 타임스탬프 + 메타 정보 요약
- 접기/펼치기 토글
- 히스토리 클리어 버튼
- 빈 상태: "아직 세션 이벤트가 없습니다."

#### 사용자 액션
- "Refresh All" 버튼 (모든 데이터 새로고침)
- 빠른 이동 링크들 ("→ Mapping 열기", "→ Teleop으로 이동")
- 히스토리 확장/축소/클리어

#### 자동 갱신
- 리소스: 5초 간격 자동 갱신
- 히스토리: 30초 간격 자동 갱신

### 4.2 디바이스 매핑

**목적**: 카메라와 로봇 팔에 고정 이름(심볼릭 링크)을 부여하여, 기기를 뽑았다 꽂아도 항상 같은 이름으로 인식되게 하는 기능.

#### 보여줄 정보

**A. 매핑 완료 상태 판정**
- 상태 뱃지: "Mapping Ready" 또는 "Mapping Incomplete"
- 매핑 체크리스트:
  - 카메라 역할 할당: X / Y 완료 (예: 2/3)
  - 팔 역할 할당: X / Y 완료
  - 중복 역할 여부: "없음" 또는 "발견됨"
  - 적용 상태: idle / applying / applied / error

**B. udev 규칙 상태**
- 설치 여부 표시:
  - 설치됨: 경로와 함께 "✓ 설치 완료"
  - 미설치 + 원클릭 가능: "⚠ 미설치. 원클릭 설치 가능" + 설치 버튼
  - 미설치 + Root 필요: 수동 설치 명령어 안내
- 상세 정보 (접기/펼치기):
  - 설치 경로, 폴백 경로
  - 현재 활성 규칙 테이블 (카메라 규칙 / 팔 규칙)
  - 각 규칙: Port, SYMLINK, MODE, STATUS(Active/Inactive)

**C. 카메라 매핑 영역**
- 안내 문구: "프리뷰는 대역폭 안전 모드(144p @ 5fps)로 실행됩니다."
- 감지된 카메라별 카드:
  - 프리뷰 영역: 클릭하면 실시간 미리보기 활성화/비활성화
  - 프리뷰 이미지 (활성화 시 MJPEG 스트림)
  - "이 카메라는 어디에 있나요?" 안내
  - 역할 드롭다운: Not used / Top Camera 1,2,3 / Wrist Camera 등
  - 메타 정보: Port, Path

**D. 팔 매핑 영역**
- 감지된 팔별 카드:
  - 디바이스 경로 (/dev/ttyUSBx)
  - 시리얼 번호
  - 역할 드롭다운: (없음) / Follower / Leader 등
  - 역할 충돌 시: 확인 대화 → 역할 교환 처리
- 빈 상태: "팔이 감지되지 않았습니다." + 다른 기능 이동 링크

**E. 팔 식별 마법사 (Identify Wizard)**
- 목적: 같은 모델의 팔이 여러 개일 때, 어떤 포트가 어떤 팔인지 식별
- 방식: "하나를 뽑았다 다시 꽂으세요" → 차이점 감지
- 표시 정보: 식별된 팔의 디바이스명, 시리얼, 커널
- 액션: "Start Identify" → 결과 나오면 → 역할 할당 "Assign" 버튼

#### 사용자 액션
- Refresh 버튼
- 원클릭 규칙 설치 (조건부)
- 카메라 역할 변경 (드롭다운)
- 팔 역할 변경 (드롭다운) — 자동으로 규칙 적용 예약
- 팔 식별 마법사 시작/취소/할당
- 카메라 프리뷰 토글

#### 자동 동작
- 역할 변경 시 자동으로 규칙 적용이 예약됨 (debounce)

### 4.3 모터 설정 및 검증

**목적**: 로봇 팔의 모터를 초기 설정하고, 개별 모터의 상태를 확인/제어하는 기능.

#### 보여줄 정보

**A. Quick Guide 카드**
- 모터 설정 절차 안내 텍스트:
  - "각 서보 모터에 고유 ID를 부여합니다. 팔당 한 번 실행하면 펌웨어에 영구 저장됩니다."
  - "콘솔에서 키보드 입력을 요구하면 하단의 글로벌 콘솔 드로어에서 입력하세요."
  - "설정 후 Step 2로 각 모터가 정상 응답하는지 검증한 뒤 캘리브레이션으로 진행하세요."

**B. Step 1: 모터 설정 실행**
- 팔 역할 타입 드롭다운 (so101_follower, so100_leader 등)
- 팔 포트 드롭다운 (감지된 팔 목록)
- 포트 없을 시 안내: "팔 포트가 감지되지 않았습니다."
- Start/Stop 버튼 (ProcessButtons)
- 완료 후: "모터 설정 완료? → 캘리브레이션으로 이동" 안내

**C. 연결된 팔 목록**
- 감지된 팔별: 상태 점, 이름, 경로

**D. Step 2: 모터 검증 (Motor Monitor)**
- 연결 제어 바:
  - 포트 셀렉터
  - Connect / Disconnect 버튼
  - Freewheel 토글 (모터 잠금/풀림)
  - E-Stop (긴급 정지) 버튼
  - 연결 상태: "포트명 · N개 모터"
- Freewheel 활성화 시: 배너로 알림
- 에러 발생 시: 에러 메시지 라인

**E. 개별 모터 카드 (연결 시 표시)**
- 각 모터별 카드:
  - 모터 ID
  - 현재 위치 (숫자 또는 에러 표시)
  - 부하(Load) 수치 + 심각도 색상
  - 전류(Current) 수치 + 심각도 색상
  - 충돌(Collision) 뱃지 (감지 시)
  - 타겟 위치 제어:
    - ▼ 버튼 (-10)
    - 숫자 입력 (0~4095)
    - ▲ 버튼 (+10)
    - Move 버튼 (타겟으로 이동)
    - Freewheel 또는 충돌 상태에서 Move 비활성화
  - 충돌 시: "Clear Collision" 버튼

#### 사용자 액션
- 팔 타입/포트 선택
- 모터 설정 Start/Stop
- 모터 모니터 Connect/Disconnect
- Freewheel 토글
- E-Stop
- 개별 모터 타겟 조정 + Move
- 충돌 클리어
- "캘리브레이션으로 이동" 네비게이션

#### 상태
- 상태 뱃지: Running / Ready / Action Needed
- 프로세스 충돌 시 차단 카드 표시
- 모터 모니터 폴링 간격: 100ms (실시간 위치/부하/전류 갱신)

### 4.4 캘리브레이션

**목적**: 로봇 팔의 관절 범위를 측정하여 캘리브레이션 파일을 생성하는 기능.

#### 보여줄 정보

**A. 모드 토글**
- Single Arm / Bi-Arm 전환

**B. Single Arm 모드**
- Step 1: 팔 선택
  - 팔 역할 타입 드롭다운
  - 팔 포트 드롭다운
  - 팔 ID 입력 (캘리브레이션 파일명)
  - 매칭 경고 (타입과 포트의 불일치 감지 시)
- 캘리브레이션 파일 상태:
  - Found / Missing / Checking 상태
  - 파일 발견 시: 수정 시간, 크기 표시
  - 파일 발견 + 실행 안 함: "→ 원격 조작으로 이동" 힌트
  - 매칭 경고: 선택한 타입과 연결된 팔이 불일치 시 "No existing {role} calibration profile matches {port}" 경고
- Start/Stop 버튼

**C. Bi-Arm 모드**
- 디바이스 타입, 결합 팔 ID
- Left Arm Port, Right Arm Port
- 캘리브레이션 순서 안내: "Both arms are calibrated sequentially in a single run."
- Start/Stop 버튼

**D. 팔 식별 패널**
- "어떤 포트가 어떤 팔인지 모르겠다면?" → Identify Wizard 열기
- 디바이스 매핑의 식별 마법사와 동일 동작

**E. 기존 캘리브레이션 파일 관리**
- 타입별 필터 드롭다운 + Refresh 버튼
- 파일 목록 (타입별 그룹: Follower / Leader / 기타)
- 각 파일: ID, 수정 시간, 삭제 버튼
- **파일 매칭 표시**: 각 파일 옆에 색상 점 — 초록(현재 연결된 팔과 매칭), 회색(매칭 안 됨)
- 파일 클릭 시: 해당 ID와 타입이 자동으로 설정 폼에 반영

**F. 실시간 모터 범위 시각화**
- 캘리브레이션 진행 중 또는 완료 후:
  - 모터별 행: 이름 라벨 + 수평 트랙 바
  - 트랙 바 위에: 범위 바 (min~max 구간 하이라이트) + 현재 위치 마커
  - 트랙 바 아래: MIN / POS / MAX 각각의 라벨과 숫자 값 표시
- 데이터 없을 시: 플레이스홀더 ("Waiting for calibration… Start process to see live ranges.")

#### 사용자 액션
- Single/Bi-Arm 모드 전환
- 팔 타입/포트/ID 선택
- Identify Wizard 실행
- 캘리브레이션 Start/Stop
- 기존 파일 삭제
- 파일 목록 새로고침
- "원격 조작으로 이동" 네비게이션

#### 상태
- 차단 카드: 캘리브레이션 불가 사유 표시 (포트 미선택, 충돌 프로세스 등)

### 4.5 원격 조작 (Teleop)

**목적**: 로봇 팔을 실시간으로 원격 조작하면서 멀티 카메라 피드를 모니터링하는 기능.
**특이사항**: 이 앱에서 가장 중요한 실시간 기능 중 하나.

#### 보여줄 정보

**A. 제어 모드 토글**
- Single Arm / Bi-Arm 전환

**B. 런타임 상태 카드**
- 팔 연결 뱃지: "ready" 또는 "missing X"
- 카메라 피드 뱃지: "N/M" (사용 가능 / 매핑된)
- 프로세스 충돌 뱃지: 충돌 사유 또는 "없음"
- 실행 중 안전 경고: "⚠️ 예상치 못한 움직임 → 즉시 Stop. 손을 치우세요."

**C. Step 1: 팔 연결 설정**
- Robot Type 드롭다운 (so101_follower 등)
- Teleoperator Type 드롭다운 (so101_leader, keyboard 등)
- Robot Capabilities 카드:
  - 기능 뱃지들: has_arms, has_wheels, has_cameras, is_remote, **has_keyboard_teleop** 등
  - has_keyboard_teleop: Keyboard 텔레옵 지원 여부 뱃지 (텔레옵 타입 선택에 영향)
  - 호환 Teleoperator 목록
- 팔 포트/ID 설정 (모드에 따라 다름):
  - Single: Follower Port/ID, Leader Port/ID
  - Bi-Arm: Left/Right Follower/Leader Port/ID
- Arm ID 안내: "캘리브레이션 프로필 파일명을 선택합니다."

**D. Step 2: 카메라 피드 설정**
- 매핑된 카메라 수 / 현재 사용 가능 수 표시
- 매핑된 카메라 행 (역할 ↔ 경로)
- 매핑 없을 시: "→ Mapping 열기" 버튼
- 고급 스트림 설정 (접기/펼치기):
  - 코덱: MJPG / YUYV
  - 해상도: 1280×720 / 800×600 / 640×480 / 320×240
  - FPS: 30
  - JPEG 품질: 30~95 슬라이더

**E. 카메라 피드 갤러리**
- 각 카메라별 타일:
  - 실시간 영상 (MJPEG 프레임)
  - "LIVE" 뱃지 오버레이
  - FPS 뱃지 (예: "30 fps")
  - 일시정지/재개 제어
  - 카메라 이름 — 경로 라벨
- 카메라 없을 시: "카메라가 감지되지 않았습니다." + "→ Mapping 열기"

**F. 실행 제어 (하단 고정)**
- 텔레옵 상태 요약: "TELEOP ACTIVE" / "Single Arm" / 속도
- 루프 성능 표시 (실행 중): "Loop: Xms (YHz)" + 색상 등급
- 속도 제어 드롭다운: 0.1x, 0.25x, 0.5x, 0.75x, 1.0x
- Start/Stop 버튼
- 차단 시: 차단 사유 표시, Start 비활성화

#### 사용자 액션
- 모드 전환 (Single/Bi-Arm)
- Robot/Teleop 타입 선택
- 팔 포트/ID 선택
- 카메라 피드 일시정지/재개
- 스트림 설정 변경
- 속도 변경
- Start/Stop Teleop
- "녹화로 이동" 링크 (실행 후 표시)

#### 상태 & 전환
- Idle → (Start) → Running → (Stop) → Idle
- Running 중: 안전 경고 표시, 피드 활성, 루프 성능 표시
- 차단: Start 비활성, 차단 사유 카드 표시

### 4.6 에피소드 녹화

**목적**: 원격 조작하면서 에피소드 데이터를 녹화하여 학습용 데이터셋을 만드는 기능.
**특이사항**: 원격 조작과 구조가 유사하지만, 녹화 계획 + 에피소드 진행 관리가 추가됨.

#### 보여줄 정보

**A. 녹화 모드 토글**
- Single / Bi-Arm

**B. Step 1: 녹화 계획**
- 에피소드 수 입력 (숫자, 기본값 50)
- Dataset Repo ID 입력 (텍스트, HF username 자동 프리필)
  - 유효성 검사: 포맷 에러 시 빨간 테두리 + 에러 메시지
- Resume 토글: 기존 데이터셋에 이어서 녹화 여부
- Push to Hub 토글: 녹화 완료 후 자동 업로드 여부
- Task Description 입력: 데이터셋에 기록되는 작업 설명

**C. Step 2: 디바이스 설정**
- 원격 조작과 동일 구조:
  - Robot Type, Teleop Type, Capabilities 카드
  - 팔 포트/ID 설정

**D. Step 3: 카메라 피드**
- 원격 조작과 동일 구조:
  - 매핑 카메라 행, 고급 스트림 설정

**E. 에피소드 진행 상태**
- 카메라 피드 그리드 (실시간 영상 + LIVE 뱃지 + FPS)
- 에피소드 진행률 바: 현재 / 목표
- 에피소드 상태 표시: Idle / Running
- 마지막 이벤트 텍스트: "Last: {이벤트}" (예: "saved episode 3", "discarded episode 4" 등)

**F. 실행 제어 (하단 고정)**
- 상태 요약 뱃지: RUNNING / READY / BLOCKED
- 진행 텍스트: "Episode X / Y" 또는 준비 상태 힌트
- Start Recording 버튼
- Force Stop 버튼 (실행 중)
- **에피소드별 제어 (실행 중에만 표시)**:
  - ✓ Save → (다음 에피소드로)
  - ✗ Discard ← (현재 에피소드 폐기)
  - ⏹ End (Esc) (녹화 종료)
- 차단 시: 차단 사유 표시
- 완료 후: "→ 데이터셋 관리로 이동" 링크

#### 사용자 액션
- 원격 조작의 모든 디바이스/카메라 설정 + 추가로:
- 에피소드 수, Repo ID, Task 입력
- Resume/Push 토글
- Start/Stop Recording
- 에피소드별: Save / Discard / End (키보드 단축키 지원)

### 4.7 데이터셋 관리

**목적**: 녹화된 데이터셋을 조회, 재생, 태그, 큐레이션하고, Hub에 업로드하거나 Hub에서 검색/다운로드하는 기능.
**특이사항**: 이 앱에서 가장 복잡한 기능 도메인. 다수의 하위 패널과 기능이 있음.

#### 보여줄 정보

**A. Hub 검색/다운로드 카드 (상단)**
- HF 토큰 관리: 상태 표시, 토큰 입력/저장/삭제
- Hub 검색:
  - 검색 입력 + 검색 버튼
  - 결과 목록: repo ID, 설명, 다운로드 수, **좋아요 수(♥)**, **태그 목록** (최대 4개), **최종 수정일**
  - 다운로드 버튼 → 진행률 바 (퍼센트 표시)
  - 다운로드 상태: 대기 / 진행 중(퍼센트) / 완료 / 실패(재시도 버튼)

**B. 좌측: 로컬 데이터셋 목록**
- 데이터셋별 항목:
  - 데이터셋 ID
  - 요약: 에피소드 수, 프레임 수, 용량
  - 수정 시간
  - 액션 메뉴: Quality 검사 / Hub Push / 삭제
- 로딩 상태: "데이터셋 로딩 중..."
- 빈 상태: "로컬 캐시에 데이터셋이 없습니다. 녹화 기능으로 녹화하거나 Hub에서 다운로드하세요."

**C. 우측: 데이터셋 상세 (선택 시)**
- 미선택 시: 빈 상태 + "최근 데이터셋 선택" / "녹화로 이동" 버튼

- **데이터셋 헤더**:
  - 데이터셋 ID
  - 통계: "N episodes · M frames · X FPS"
  - 카메라 칩 목록
  - 액션: Quality 검사 / Hub Push / 삭제

- **Hub 업로드 상태** (활성 시):
  - 상태 라벨 + 진행률 바 + 상세 메시지
  - 에러 시 재시도 버튼

- **품질 검사 패널** (실행 후):
  - 전체 품질 점수 (숫자)
  - 개별 검사 항목 리스트:
    - 각 항목: 검사 이름(name), 심각도 레벨(ok/warn/fail + 색상 점), 상세 메시지(message)
  - 점수 60 이상 시: "→ AI 학습으로 이동" 버튼

**D. 에피소드 재생기**
- 에피소드 셀렉터 드롭다운: "Episode X (Y frames)"
- 에피소드 페이지 네비게이션: 이전/다음 + 페이지 표시
- 에피소드 태그 뱃지: good / bad / review / untagged

- **비디오 그리드**: 카메라별 동기화된 비디오 타일
  - 카메라명 라벨
  - 비디오 없을 시: "이 데이터셋에 비디오 데이터가 없습니다."

- **비디오 컨트롤 바**:
  - 현재 시간 표시
  - 타임라인 스크러버 (range slider)
  - 전체 시간 표시
  - ⏮ 이전 에피소드 / ▶ Play / ⏸ Pause / ⏭ 다음 에피소드
  - 재생 속도 드롭다운: 0.25x, 0.5x, 1x, 1.5x, 2x

- **에피소드 필터 & 검색**:
  - 에피소드 번호 검색 입력
  - 태그 필터 드롭다운: 전체 / good / bad / review / untagged

**E. 에피소드 태깅**
- 태그 버튼: 👍 Good / 👎 Bad / 🔍 Review / ✕ Clear
- Auto-next 토글: 태그 후 자동으로 다음 에피소드로

**F. 자동 플래그 패널 (Auto-Flag)**
- 목적: 품질 낮은 에피소드를 자동 탐지
- 에피소드 통계 로드 버튼
- **↻ Recompute 버튼** (통계 재계산 — 캐시 무효화 후 재실행)
- 통계 진행률 (로딩 중)
- 로드 완료 시:
  - 요약: 에피소드 수, 프레임 수, jerk 등
  - 프리셋: Strict / Balanced / Lenient
  - 임계값 제어: 최소 프레임, 최소 움직임, 최대 jerk, 최대 jerk 비율
  - 각 기준별 토글 스위치
  - 플래그된 에피소드 목록 + 사유
  - 일괄 태그 버튼: "플래그된 에피소드를 Bad로 태그"

**G. 데이터셋 파생 (Curation/Derive)**
- 파생 모드 선택: 현재 필터 / Good만 / Bad 제외
- 유지/제거 건수 요약
- 새 데이터셋 Repo ID 입력
- CLI 미리보기 (접기/펼치기)
- "파생 데이터셋 생성" 버튼
- 작업 진행 상태: 진행률, 로그, 취소 버튼
- 완료/실패/재시도

#### 사용자 액션
- 데이터셋 선택
- Hub 검색/다운로드
- Hub Push
- Quality 검사
- 데이터셋 삭제
- 에피소드 재생/탐색
- 에피소드 태깅 (개별 + 일괄)
- 자동 플래그 실행 + 임계값 조정
- 데이터셋 파생 생성
- Hub 토큰 관리

### 4.8 AI 학습 (Training)

**목적**: 녹화된 데이터셋으로 AI 정책(Policy)을 학습시키고, 실시간으로 학습 진행을 모니터링하는 기능.

#### 보여줄 정보

**A. 학습 설정 카드**
- Policy Type 드롭다운: ACT / Diffusion Policy / TD-MPC2
- Dataset Source 토글: Local / Hugging Face
  - Local 선택 시: 로컬 데이터셋 드롭다운
    - 데이터셋 없을 시: 안내 배너 + 소스 전환 제안
  - Hugging Face 선택 시: Repo ID 텍스트 입력
- Training Steps:
  - 프리셋 버튼: Quick (1K), Standard (50K), Full (100K)
  - 스텝 수 숫자 입력
- 고급 오버라이드 (접기/펼치기):
  - Learning Rate 입력
  - Model Output Repo 입력 (선택)
- Compute Device 드롭다운: CUDA (GPU) / CPU / MPS (Apple Silicon)

**B. CUDA Preflight 및 PyTorch 설치**
- Preflight 검사 실패 시:
  - 경고 메시지 + 사유
  - 자동 대응 액션:
    - "Install CUDA PyTorch (Nightly)" 버튼 (CUDA 문제 시)
    - 또는: 수정 명령어 표시 + "Run Fix" 버튼
    - 실행 중: 진행 상태 + Stop 버튼

**C. 학습 진행 영역**
- 상태 뱃지: RUNNING / IDLE / BLOCKED
- 진행률 바 (수평, 퍼센트)
- 실시간 수치:
  - Step: 현재 / 전체
  - Loss: 최신 값
  - ETA: 예상 남은 시간
- 신호 없을 시: "아직 학습 신호 없음..."

**D. Loss 트렌드 차트**
- 캔버스 기반 실시간 차트
- X축: 스텝, Y축: Loss 값
- 데이터 없을 시: "데이터 없음 — 학습 중 loss 값이 여기에 표시됩니다."

**E. 체크포인트 패널**
- Refresh 버튼
- 체크포인트 목록: 이름/경로
- 로딩 상태 / 타임아웃 + 재시도
- 빈 상태: "체크포인트 없음. 먼저 모델을 학습하세요."
- 체크포인트 존재 시: "→ 정책 평가로 이동" CTA

**F. GPU 상태 패널**
- GPU 사용률 + 바
- VRAM 사용량 + 바
- 사용 불가 시: "NVIDIA GPU 정보를 사용할 수 없습니다."
- OOM 감지 시 (학습 완료 후):
  - 메시지: "GPU 메모리 부족. 현재 배치 크기: X. Y로 줄이고 재시도할까요?"
  - "Reduce & Retry" 버튼

**G. 실행 제어 (하단 고정)**
- 상태 요약: STARTING / RUNNING / READY / BLOCKED
- Start/Stop 버튼
- **Colab 관련 기능**:
  - "Train on Colab" 버튼: 설정을 HF Hub에 업로드 + Colab 노트북을 새 탭에서 열기
  - "Copy Colab Snippet" 버튼: 학습 코드 스니펫을 클립보드에 복사 (Colab 첫 셀에 붙여넣기용)
  - Colab 안내 메시지: "Colab 세션은 영구적이지 않음 — 유휴 시 연결 해제 가능, 무료 런타임 시간 제한"
  - Colab Device 자동 보정: MPS 선택 시 Colab에서는 자동으로 CUDA 사용
  - Model Output Repo: Colab 체크포인트 업로드 대상 저장소

#### 사용자 액션
- 정책/데이터셋/디바이스 설정
- 프리셋 선택
- 고급 파라미터 조정
- CUDA 문제 해결 (설치/수정)
- Start/Stop Training
- 체크포인트 Refresh
- OOM 시 Reduce & Retry
- "Eval로 이동"

### 4.9 정책 평가 (Eval)

**목적**: 학습된 AI 정책을 실제 로봇 또는 시뮬레이션에서 평가하는 기능.

#### 보여줄 정보

**A. 평가 설정 패널 (EvalConfigPanel)**
- Policy Source: 경로 선택 또는 체크포인트에서 자동 연결
- Dataset Repo ID
- 에피소드 수 입력
- Device 드롭다운: CUDA / CPU
- Env Type 드롭다운: 사용 가능한 Gym 환경 목록
  - 각 항목: 타입명, 라벨, 설치 여부
  - 미설치 시: 경고 메시지 ("{module} is not installed") + **"Install {module}" 버튼** (pip install 실행)
  - 설치 진행 중: "Installing..." 상태 + "Stop Install" 버튼
  - env metadata 없을 시: "No env metadata found. For HF or real-robot policies, select 'gym_manipulator'." 가이드
- Task 입력
- Dataset Override 설정

**B. 로봇 카메라 설정 (EvalRobotConfig)**
- **조건부 표시**: Env Type이 `gym_manipulator`일 때만 표시 (실제 로봇 평가용)
- 실제 로봇 평가 시: 카메라 매핑 설정 UI
- 원격 조작의 카메라 설정과 유사한 구조

**C. 평가 진행 패널 (EvalProgressPanel)**
- 실행 상태 및 진행률
- 에피소드별 결과:
  - Final Reward
  - Best / Worst Episode
  - 에피소드 카운트
- 요약 통계

**D. 실행 제어**
- Start/Stop Eval
- 차단 사유 표시 (차단 시)

#### 사용자 액션
- 정책 소스/데이터셋/환경 설정
- 카메라 매핑 설정 (실제 로봇 시)
- Start/Stop Eval
- Gym 환경 자동 설치 트리거
- "AI 학습으로 이동" 링크

## 5. 공통 UI 패턴 정리

디자이너가 일관성 있게 반복 사용해야 할 패턴들:

### 5.1 차단 카드 (Blocker Card)
- 특정 조건 미충족 시 표시되는 경고 카드
- 차단 사유 목록 (칩/뱃지 형태)
- 각 사유별 해결 액션 링크
- 관련 기능: 원격 조작, 녹화, AI 학습, 정책 평가, 캘리브레이션

### 5.2 ProcessButtons (Start/Stop 제어)
- 모든 프로세스 실행 기능에서 사용
- Start 버튼 (라벨 커스터마이즈 가능)
- Stop 버튼 (실행 중에만)
- 충돌 힌트 (다른 프로세스가 같은 하드웨어 사용 중일 때)
- 관련 기능: 모터 설정, 캘리브레이션, 원격 조작, 녹화, AI 학습, 정책 평가

### 5.3 상태 뱃지 (Status Verdict)
- 화면 헤더 우측에 표시
- Running / Ready / Action Needed / Blocked
- 색상: 초록(ready) / 노랑(warn) / 빨강(error)
- 관련: 모든 기능 도메인

### 5.4 Robot Capabilities 카드
- 선택된 로봇의 기능을 뱃지로 표시
- has_arms, has_wheels, has_cameras, is_remote, **has_keyboard_teleop** 등
- has_keyboard_teleop 뱃지: 키보드 텔레옵 지원 여부 ("Keyboard" 라벨)
- 호환 Teleoperator 목록
- 관련 기능: 원격 조작, 녹화

### 5.5 MappedCameraRows
- 매핑된 카메라 역할 ↔ 경로 행 목록
- 매핑 없을 시: 디바이스 매핑 기능으로 이동 힌트
- 관련 기능: 원격 조작, 녹화

### 5.6 카메라 피드 갤러리
- 실시간 MJPEG 프레임 이미지 타일
- LIVE 뱃지, FPS 뱃지
- 일시정지/재개
- 관련 기능: 원격 조작, 녹화

### 5.7 스티키 컨트롤 바 (하단 고정)
- 실행 요약 + 핵심 액션 버튼
- 항상 화면 하단에 고정
- 관련 기능: 원격 조작, 녹화, AI 학습

## 7. 레퍼런스 앱

아래 3개 앱을 참고하여 와이어프레임의 시각적 방향을 잡아주세요.
각 앱에서 **어떤 부분을 참고해야 하는지**를 구체적으로 명시합니다.

---

### 7.1 Foxglove Studio — 로봇 데이터 시각화 도구

- **확인 방법**: https://app.foxglove.dev (무료 가입, 브라우저에서 즉시 체험 가능)
- **추가 자료**: https://foxglove.dev 메인 페이지 데모 영상, https://foxglove.dev/blog 에서 UI 스크린샷 다수

**참고할 패턴:**

| Foxglove 패턴 | LeStudio 적용 대상 |
|---|---|
| 멀티 패널 레이아웃 (카메라 + 차트 + 데이터를 동시에 배치) | 원격 조작: 카메라 피드 갤러리 + 상태 정보 동시 표시 |
| 실시간 카메라 피드 타일 (LIVE 오버레이, 카메라명 라벨) | 원격 조작, 녹화: 멀티카메라 실시간 피드 |
| 타임라인 스크러빙 (하단 타임바 + 재생 컨트롤) | 데이터셋 관리: 에피소드 비디오 재생기 |
| 다크 테마 기반 프로페셔널 톤 | 앱 전체의 다크 모드 기본 톤 |
| 사이드바 + 메인 콘텐츠 분리 구조 | 앱 쉘 전체 레이아웃 |

**참고하지 말아야 할 점:**
- Foxglove는 "데이터 뷰어"에 특화되어 있어, 설정 폼/워크플로우 UI가 약함
- 사용자가 직접 패널을 자유 배치하는 구조 → LeStudio는 구조화된 레이아웃이므로 이 부분은 따라가지 않음

---

### 7.2 Weights & Biases (wandb) — ML 실험 추적 대시보드

- **확인 방법**: https://wandb.ai (무료 가입, 즉시 대시보드 체험 가능)
- **UI 문서**: https://docs.wandb.ai/guides/app — Panels, Line Plots, Run Comparer 등 상세 설명 + 스크린샷

**참고할 패턴:**

| wandb 패턴 | LeStudio 적용 대상 |
|---|---|
| 실시간 Line Plot (Loss/LR 곡선, X축 스텝, Y축 메트릭) | AI 학습: Loss 트렌드 차트 |
| Run 진행률 표시 (현재 스텝/전체, ETA, 진행률 바) | AI 학습: 학습 진행 영역 |
| Run 비교 테이블 (하이퍼파라미터 diff) | AI 학습: 프리셋 비교 (Quick/Standard/Full) |
| 워크스페이스 레이아웃 (좌측 Run 목록 + 우측 차트/메트릭) | AI 학습: 좌측 설정 + 우측 차트/체크포인트 |
| 체크포인트/아티팩트 브라우저 | AI 학습: 체크포인트 패널 |
| 패널별 Refresh + 로딩/타임아웃 상태 처리 | 전체 앱: 데이터 로딩 패턴 |

**참고하지 말아야 할 점:**
- wandb는 순수 소프트웨어 도구라 하드웨어 상태 표시(포트, 모터, 카메라 연결)에 대한 패턴이 없음
- 다수 실험 비교가 핵심인 wandb와 달리 LeStudio는 단일 실행 모니터링이 중심

### 7.3 Home Assistant — IoT/스마트홈 대시보드

- **확인 방법**: https://demo.home-assistant.io (라이브 데모, 가입 불필요)
- **커뮤니티 대시보드**: "Home Assistant Mushroom dashboard" 이미지 검색 시 고품질 커스텀 대시보드 다수
- **테마 갤러리**: "Home Assistant dashboard themes 2025" 검색

**참고할 패턴:**

| Home Assistant 패턴 | LeStudio 적용 대상 |
|---|---|
| 디바이스 상태 카드 (연결됨/끊김/경고, 색상 점, 간결한 라벨) | 시스템 상태: 카메라/팔/프로세스 상태 카드 |
| 카드 그리드 레이아웃 (반응형, auto-fill) | 시스템 상태, 모터 설정: 카드 그리드 |
| 실시간 수치 표시 (게이지, 바, 퍼센트) | 시스템 상태: CPU/RAM/Disk 리소스 바 |
| 다크/라이트 테마 전환 (CSS 변수 기반) | 앱 전체: 테마 시스템 |
| 센서 그룹별 카드 분류 (방별, 타입별) | 디바이스 매핑: 카메라/팔 그룹 분류 |
| 토글/스위치 컨트롤 (on/off 제어) | 모터 설정: Freewheel 토글, E-Stop |

**참고하지 말아야 할 점:**
- Home Assistant는 "홈 자동화" 도구라 순차적 워크플로우(Setup → Operate → Data → ML) 개념이 없음
- 커뮤니티 테마가 너무 다양하므로, Mushroom 카드 스타일의 미니멀 다크 테마를 기준으로 볼 것
- 모바일 퍼스트 성향이 강함 → LeStudio는 데스크톱 퍼스트이므로 정보 밀도를 더 높여도 됨

## 9. 현재 UI의 핵심 불만 사항

현재 와이어프레임/UI에서 가장 크게 느끼는 문제:

> **레이아웃 밀도 배분이 잘못되어 있다.**
> - 넓어야 할 요소(카메라 피드, 차트, 타임라인 등)가 충분한 공간을 차지하지 못함
> - 앵커(고정 영역)가 되지 않아도 되는 요소가 불필요하게 큰 공간을 차지함
> - 반대로 집약적이어야 할 컨트롤/설정 영역은 집약적이지 않고 흩어져 있음

**디자이너에게 요청하는 방향:**
- 각 기능의 **주인공 콘텐츠**(카메라 피드, 차트, 데이터 테이블 등)가 화면의 대부분을 차지하도록 설계
- 설정/컨트롤은 사이드 패널, 콜랩서블 섹션, 또는 모달로 집약하여 주 콘텐츠를 방해하지 않도록
- "큰 건 크게, 작은 건 작게" — 정보 위계(hierarchy)가 시각적 크기에 정확히 반영되어야 함
- §6 복잡도 매트릭스를 참고하여, 복잡한 기능일수록 레이아웃 밀도 설계에 더 신경 쓸 것
